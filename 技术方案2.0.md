# 新生报到系统 2.0 技术方案书

# 一、项目概述
## 1.1 升级目标
在 1.0 版本"点击报到"极简链路的基础上，**增加新生信息采集功能**。让小程序和网页端不仅能报到，还能让新生填写个人基本资料，并将这些信息持久化存储，管理员可在后台统一查看。

**一句话目标**：新生报到时，先填表，再提交，报到记录中包含姓名、学号等信息。

## 1.2 功能范围界定
### 必做功能
- ✅ 个人信息填写表单（姓名、学号、手机号）
- ✅ 表单数据存入数据库，与报到记录关联
- ✅ 网页版和小程序端同步更新，支持填表后报到
- ✅ 管理后台优化，支持查看和搜索学生信息

### 不实现功能
- ❌ 微信登录（采用匿名标识代替用户身份）
- ❌ 图片上传（身份证、录取通知书等）
- ❌ 管理员审核流程
- ❌ 宿舍分配功能

## 1.3 验收标准
1. 网页端和小程序端均能显示表单页面
2. 缺少任一字段点击提交时，前端给出提示，不发送请求
3. 正确填写三个字段后提交，收到成功响应
4. 后台管理界面中，学生信息表有对应记录，报到记录表有一条关联数据
5. 重复提交时，学号唯一性可被约束（数据库层面或后端校验）

# 二、系统架构设计
## 2.1 总体架构
### 2.1.1 架构模式
采用经典三层架构，在 1.0 基础上扩展数据采集能力：

**表现层**：微信小程序 + 网页端，提供表单输入界面与提交按钮。

**业务逻辑层**：Django 框架，编写数据接收、校验、存储接口，处理学生信息与报到记录的关联。

**数据持久层**：SQLite 数据库，建立学生信息表与报到记录表，通过一对一关联确保数据完整性。

# 三、技术实现方案
## 3.1 数据库设计
### 3.1.1 数据模型关系
本阶段存在两张数据表，通过一对一关联：

```
学生信息表 (StudentInfo)  1 : 1  报到记录表 (RegistrationRecord)
```

### 3.1.2 核心表结构说明

#### 表1：学生信息表 (StudentInfo)
| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | IntegerField | 主键，自增 | 唯一标识每个学生 |
| name | CharField | 必填，最大长度50 | 学生姓名 |
| student_id | CharField | 必填，唯一，最大长度20 | 学号，不可重复 |
| phone | CharField | 必填，最大长度11 | 手机号 |
| created_at | DateTimeField | 自动填充 | 记录创建时间 |

#### 表2：报到记录表 (RegistrationRecord)
| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| id | IntegerField | 主键，自增 | 唯一标识每条报到记录 |
| student | OneToOneField | 必填，关联StudentInfo | 关联的学生信息 |
| register_time | DateTimeField | 自动填充 | 报到时间 |

**设计说明**：
- 采用一对一关联而非扩展原有表，保持数据结构清晰
- 学号字段设置唯一约束，防止重复提交
- 时间字段自动填充，减少前端传参

## 3.2 接口设计
### 3.2.1 核心接口
- **接口地址**：`/api/register_with_info/`
- **请求方法**：POST
- **功能描述**：接收学生个人信息，创建学生信息记录与报到记录，返回操作结果

**请求参数**（JSON 格式）：
```json
{
  "name": "张三",
  "student_id": "2024001001",
  "phone": "13800138000"
}
```

**返回数据**（成功）：
```json
{
  "success": true,
  "message": "报到成功",
  "data": {
    "name": "张三",
    "student_id": "2024001001",
    "phone": "13800138000",
    "register_time": "2024-09-01 10:30:00"
  }
}
```

**返回数据**（失败）：
```json
{
  "success": false,
  "message": "学号已存在，请勿重复报到"
}
```

### 3.2.2 接口处理流程
```
接收请求 → 校验字段完整性 → 检查学号唯一性 → 
创建学生信息记录 → 创建报到记录并关联 → 返回成功响应
        ↓
    任一环节失败 → 返回错误提示
```

## 3.3 前端实现方案
### 3.3.1 网页版前端
**文件位置**：`网页版/index.html`

**改动内容**：
1. 移除原有单一"点击报到"按钮
2. 添加表单 HTML 结构：
   - 姓名输入框（type="text"）
   - 学号输入框（type="text"）
   - 手机号输入框（type="tel"）
   - 提交按钮
3. 使用 JavaScript 实现：
   - 表单非空校验
   - 手机号格式校验（可选）
   - 通过 `fetch` 发送 POST 请求
   - 处理响应并展示成功/失败信息

**交互流程**：
```
页面加载 → 显示空白表单 → 用户填写 → 点击提交 → 
前端校验 → 发送请求 → 显示结果 → 成功则清空表单
```

### 3.3.2 小程序前端
**文件位置**：`小程序/pages/index/`

**改动内容**：
1. 修改 `index.wxml`：
   - 将原有按钮替换为 `<input>` 组件
   - 添加表单容器与提交按钮
2. 修改 `index.js`：
   - 添加 `data` 字段绑定表单数据
   - 实现 `bindinput` 事件处理函数
   - 为提交按钮绑定事件处理函数
   - 调用 `wx.request` 发送表单数据
   - 处理回调，展示 `wx.showToast` 提示

**交互流程**：
```
页面加载 → 显示空白表单 → 用户填写 → 点击提交 → 
前端校验 → wx.request 发送请求 → wx.showToast 显示结果
```

## 3.4 管理后台优化
### 3.4.1 注册模型
在 `api/admin.py` 中注册 `StudentInfo` 模型，配置列表显示字段。

### 3.4.2 功能支持
- 列表视图显示：姓名、学号、手机号、报到时间
- 搜索功能：按学号搜索
- 排序功能：按报到时间排序（默认倒序）
- 详情查看：点击记录查看完整信息

## 3.5 安全设计
### 3.5.1 开发阶段
- 暂时关闭 Django 的跨站请求伪造（CSRF）保护，便于小程序调试
- 不进行数据加密，仅用于本地开发验证

### 3.5.2 输入校验
- 后端校验：必填字段非空检查、学号唯一性检查、手机号格式检查
- 前端校验：非空检查、基础格式检查
- 双重校验确保数据完整性

### 3.5.3 数据保护
- 学号唯一约束防止重复提交
- 数据库层面约束与后端逻辑校验相结合

## 3.6 性能优化
- 单次请求执行两次数据库插入（学生信息 + 报到记录），响应时间极短
- 学号字段添加唯一索引，加速重复检查
- 报到时间字段添加索引，支持后台快速排序
- Django 开发服务器满足调试阶段并发需求

# 四、运维部署方案
## 4.1 开发环境
| 配置项 | 要求 |
|--------|------|
| 操作系统 | Windows 10 或 macOS |
| Python 版本 | 3.10 及以上 |
| 微信开发者工具 | 最新稳定版 |
| 代码编辑器 | VS Code（推荐） |
| Django 版本 | 4.2 及以上 |

## 4.2 本地部署流程
1. 确认 1.0 版本项目可正常运行
2. 备份当前 SQLite 数据库文件（复制 `db.sqlite3`）
3. 在 `api/models.py` 中新建 `StudentInfo` 模型
4. 修改 `RegistrationRecord` 模型，添加与 `StudentInfo` 的一对一关联
5. 执行 `python manage.py makemigrations` 生成迁移文件
6. 执行 `python manage.py migrate` 应用迁移
7. 在 `api/views.py` 中编写新的视图函数
8. 在 `api/urls.py` 中添加新接口路由
9. 在 `api/admin.py` 中注册新模型
10. 更新 `网页版/index.html`，添加表单与提交逻辑
11. 更新小程序页面文件，替换按钮为表单组件
12. 启动 Django 开发服务器
13. 在浏览器中测试网页版表单提交
14. 在微信开发者工具中测试小程序表单提交
15. 登录 Django Admin，验证数据是否正确存储

## 4.3 调试注意事项
- 小程序开发期间保留"不校验合法域名"选项
- 前端请求地址需根据实际启动端口调整（如 `http://127.0.0.1:8000`）
- 使用浏览器开发者工具查看网络请求与响应
- 使用 Django 终端日志查看后端处理过程

## 4.4 后续生产环境建议
- 数据库切换为 PostgreSQL，提升并发处理能力
- 采用 Nginx + Gunicorn 部署 Django 应用
- 申请 SSL 证书，启用 HTTPS 加密通信
- 接入微信登录，获取真实用户标识
- 增加图片上传功能，支持身份证、录取通知书上传
- 实现管理员审核流程，完善业务闭环

# 六、风险评估与应对
| 风险描述 | 应对措施 |
|----------|----------|
| 数据库迁移失败 | 提前备份 db.sqlite3，删除迁移文件后重新生成 |
| 学号唯一性约束冲突 | 后端捕获 IntegrityError，返回友好错误提示 |
| 小程序请求发送失败 | 确认已勾选域名校验豁免，核对接口地址与端口 |
| 跨域问题（CORS） | 开发阶段关闭 CSRF，生产环境配置 CORS 中间件 |
| 前端表单校验遗漏 | 后端必须进行二次校验，不依赖前端校验 |
| 原有 1.0 数据丢失 | 迁移前完整备份数据库，迁移后验证数据完整性 |

# 十、项目心得体会
## 核心收获
通过 2.0 版本的升级，我深刻认识到系统迭代的核心在于"在已有链路上增加价值"。1.0 版本验证了数据通路的可行性，2.0 版本则在此基础上赋予了数据实际意义——从单纯的时间戳记录升级为包含学生个人信息的完整报到记录。这种渐进式开发模式让我能够在每个阶段都获得可验证的成果，避免了"一次性开发所有功能"带来的高风险。

## 设计反思
本次升级采用"新建学生信息表 + 一对一关联"的设计，而非直接扩展原有报到记录表。这样做的好处是数据结构清晰、职责分明，未来扩展也更加灵活。但同时也要注意到，这增加了数据库查询的复杂度，需要在后续版本中考虑查询优化。

前端方面，网页版和小程序版同步更新确保了用户体验的一致性，但也带来了双倍的开发工作量。未来可考虑使用跨平台框架（如 uni-app）来减少重复开发。

## 未来展望
2.0 版本完成了从"能报到"到"有信息"的跨越，为后续的功能扩展奠定了坚实的数据基础。在 3.0 版本中，我将引入微信登录解决身份认证问题，增加图片上传完善信息采集维度，实现审核流程构建业务闭环。随着功能的不断完善，系统将逐步演进为一套真正能够服务高校新生报到场景的智慧迎新平台。
