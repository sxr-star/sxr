# 新生报到系统 4.0 - 数据库设计文档

## 一、设计概述

### 1.1 设计目标

本数据库设计旨在支持新生报到系统 4.0 版本的核心功能：
- 手机号验证码登录
- 身份证正反面照片上传与存储
- 学生信息采集与管理
- 管理员审核流程（待审核/审核通过/审核驳回）
- 宿舍号分配
- 报到数据统计看板

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| 规范化 | 遵循数据库三大范式，确保数据完整性 |
| 高性能 | 通过索引优化查询性能 |
| 可扩展 | 预留未来功能扩展空间 |
| 安全性 | 敏感数据安全存储 |

### 1.3 版本历史

| 版本 | 日期 | 变更说明 |
|------|------|----------|
| 1.0 | 2026-04-24 | 初始版本，仅包含操作记录表 |
| 2.0 | 2026-04-27 | 新增学生信息表和报到记录表 |
| 3.0 | 2026-05-19 | 新增验证码记录表，扩展学生信息表 |
| 4.0 | 2026-05-19 | 新增审核状态、驳回原因、宿舍号字段 |

## 四、数据表详细设计

### 4.1 验证码记录表（api_verificationcode）

#### 4.1.1 表结构

| 字段名 | 数据类型 | 约束 | 默认值 | 说明 |
|--------|----------|------|--------|------|
| id | Integer | PK, AUTO | - | 自增主键 |
| phone | CharField(11) | NOT NULL | - | 手机号 |
| code | CharField(4) | NOT NULL | - | 4位验证码 |
| created_at | DateTimeField | NOT NULL | auto_now_add | 创建时间 |
| is_used | BooleanField | NOT NULL | False | 是否已使用 |
| expires_at | DateTimeField | NOT NULL | - | 过期时间 |

#### 4.1.2 字段设计说明

**id**
- 类型：自增整数
- 约束：主键
- 设计考量：系统自动生成的唯一标识符，用于内部关联

**phone**
- 类型：11位字符串
- 约束：非空
- 设计考量：存储中国大陆手机号，作为查询验证码的主要条件

**code**
- 类型：4位字符串
- 约束：非空
- 设计考量：存储4位数字验证码，固定长度便于校验

**created_at**
- 类型：日期时间
- 约束：非空，自动填充
- 设计考量：记录验证码发送时间，用于计算有效期

**is_used**
- 类型：布尔值
- 约束：非空，默认False
- 设计考量：标记验证码是否已使用，防止重复使用

**expires_at**
- 类型：日期时间
- 约束：非空
- 设计考量：记录验证码过期时间，用于快速判断是否有效

#### 4.1.3 索引设计

| 索引名 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| idx_phone | phone | 普通索引 | 加速按手机号查询 |
| idx_expires_at | expires_at | 普通索引 | 加速过期数据清理 |

## 五、反范式设计考量

### 5.1 设计决策

本数据库设计遵循第三范式，但在以下方面进行了权衡：

| 设计点 | 规范化设计 | 实际设计 | 考量 |
|--------|-----------|----------|------|
| 手机号存储 | 单独建表关联 | 直接存储在StudentInfo | 查询效率优先 |
| 身份证照片 | 单独建表 | 直接存储路径 | 简化查询逻辑 |
| 审核状态 | 单独建表 | 直接存储在StudentInfo | 简化查询，提高统计效率 |

### 5.2 权衡分析

**手机号重复存储**
- 问题：VerificationCode和StudentInfo都存储手机号，存在数据冗余
- 理由：验证码表是临时表，学生信息表是永久表，分开存储更清晰
- 影响：数据一致性需要业务逻辑保证，删除学生信息不影响历史验证码记录

**身份证照片路径存储**
- 问题：直接存储路径而非二进制数据
- 理由：文件系统更适合存储大文件，数据库只存储路径便于管理
- 影响：需要确保文件路径与数据库记录同步

**审核状态冗余存储**
- 问题：审核状态直接存储在学生信息表中
- 理由：审核状态与学生信息强关联，分开存储会增加查询复杂度
- 影响：统计看板可直接查询，无需关联其他表

## 八、性能优化

### 8.1 索引优化

| 优化项 | 说明 |
|--------|------|
| 验证码查询 | phone字段索引加速查询 |
| 学号查询 | student_id唯一索引保证唯一性 |
| 过期清理 | expires_at索引加速过期数据删除 |
| 审核状态筛选 | review_status索引加速按状态查询 |

### 8.2 查询优化

```python
# 推荐查询方式
# 验证码查询
VerificationCode.objects.filter(
    phone=phone,
    is_used=False,
    expires_at__gt=datetime.now()
)

# 学生信息查询（预加载关联）
StudentInfo.objects.select_related('registrationrecord').get(student_id=student_id)

# 按审核状态筛选
StudentInfo.objects.filter(review_status='pending')

# 统计看板查询
total = StudentInfo.objects.count()
pending = StudentInfo.objects.filter(review_status='pending').count()
approved = StudentInfo.objects.filter(review_status='approved').count()
rejected = StudentInfo.objects.filter(review_status='rejected').count()
pass_rate = (approved / total * 100) if total > 0 else 0
```

### 8.3 数据清理策略

| 任务 | 频率 | 说明 |
|------|------|------|
| 清理过期验证码 | 每日 | 删除expires_at < 当前时间的记录 |
| 清理测试数据 | 按需 | 删除测试环境的学生记录 |
