# 新生报到系统 3.0 - 数据库设计文档

## 1. 文档概述

本文档描述新生报到系统 3.0 版本的数据库设计方案，详细说明数据库表结构、字段定义、索引设计以及 ER 图关系。

### 1.1 设计目标

- 实现手机号验证码登录功能的数据存储
- 实现新生个人信息采集与身份证照片上传的数据持久化存储
- 支持学生信息表与报到记录表的一对一关联
- 支持验证码记录的管理和查询
- 保证学号唯一性，防止重复报到

### 1.2 设计原则

- 遵循数据库设计三大范式
- 字段命名规范统一
- 充分考虑查询性能
- 采用表分离设计，职责分明
- 为未来扩展预留空间

### 1.3 版本说明

| 版本 | 日期 | 说明 |
|------|------|------|
| 1.0 | 2026-04-24 | 初始版本，仅包含操作记录表 |
| 2.0 | 2026-04-27 | 新增学生信息表，实现一对一关联 |
| 3.0 | 2026-04-29 | 新增验证码记录表，扩展学生信息表 |

## 3. 数据表设计

### 3.1 验证码记录表（api_verificationcode）

#### 3.1.1 表结构说明

表名：`api_verificationcode`

用途：存储手机号验证码的发送记录，包括验证码内容、发送时间、使用状态和过期时间。

## 4. ER 图

### 4.1 实体关系图

```
┌─────────────────────────┐
│   验证码记录表           │
│   (VerificationCode)    │
├─────────────────────────┤
│ id (PK)                 │
│ phone                   │
│ code                    │
│ created_at              │
│ is_used                 │
│ expires_at              │
└─────────────────────────┘
         │
         │ 查询条件
         ↓
┌─────────────────────────┐       ┌─────────────────────────┐
│   学生信息表             │       │   报到记录表             │
│   (StudentInfo)         │◄─────►│   (RegistrationRecord)  │
├─────────────────────────┤ 1:1   ├─────────────────────────┤
│ id (PK)                 │       │ id (PK)                 │
│ name                    │       │ student (FK)            │
│ student_id (UNIQUE)     │       │ registered_at           │
│ phone                   │       └─────────────────────────┘
│ id_card_photo           │
│ created_at              │
└─────────────────────────┘
```

### 4.2 关系说明

| 关系 | 说明 |
|------|------|
| 验证码记录表 → 学生信息表 | 通过手机号关联，验证码表作为查询条件 |
| 学生信息表 ↔ 报到记录表 | 一对一关联，一个学生对应一条报到记录 |

## 7. 性能优化建议

### 7.1 索引优化

- 为验证码表的 phone 字段添加索引，加速按手机号查询
- 为学生信息表的 student_id 字段设置唯一约束
- 为学生信息表的 phone 字段添加索引，加速按手机号查询

### 7.2 查询优化

- 验证码查询使用 `filter(phone=phone, is_used=False, expires_at__gt=now)` 条件
- 学生信息查询使用 `select_related` 预加载关联的报到记录
- 定期清理过期的验证码记录

### 7.3 存储优化

- 身份证照片按日期分类存储：`id_cards/年/月/日/`
- 定期清理测试数据，避免占用过多磁盘空间
- 生产环境建议将媒体文件迁移至云存储服务

## 8. 安全设计

### 8.1 数据完整性

- 学号字段设置唯一约束，防止重复提交
- 手机号字段设置非空约束，确保联系方式有效
- 身份证照片字段设置非空约束，确保身份证明材料完整

### 8.2 数据保密性

- 验证码记录表不对外暴露，仅用于后端校验
- 身份证照片文件存储在服务器本地，避免直接访问
- 生产环境建议对敏感数据进行加密存储

### 8.3 数据可用性

- 验证码设置过期时间，避免无效数据堆积
- 报到记录与学生信息建立一对一关联，确保数据一致性
- 定期备份数据库，防止数据丢失
