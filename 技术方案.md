# 新生报到系统 3.0 技术方案书

# 一、项目概述
## 1.1 升级目标
在 2.0 版本"个人信息采集"的基础上，同时增加手机号验证码登录和身份证照片上传两个功能。新生在报到前需先通过手机号验证身份，填表时需上传身份证照片作为身份证明材料。系统将手机号与学生身份绑定，身份证照片保存在服务器本地，管理员可在后台查看。

**一句话目标**：学生先用手机号验证登录，然后填表、上传身份证照片、提交报到。

## 1.2 功能范围界定
### 必做功能
- ✅ 手机号验证码发送（开发阶段用控制台打印模拟）
- ✅ 验证码校验与登录状态保持
- ✅ 身份证照片上传（支持常见图片格式，限制大小）
- ✅ 手机号与学生信息自动关联
- ✅ 保留 2.0 已有的文本信息采集功能（姓名、学号、手机号）

### 不实现功能
- ❌ 管理员审核流程
- ❌ 宿舍分配功能

## 1.3 验收标准
1. 输入手机号点击发送，后端控制台能看到生成的验证码
2. 60 秒内重复点击发送按钮被正确拦截
3. 输入正确验证码后登录成功，跳转到报到页
4. 输入错误或过期验证码，系统给出明确错误提示
5. 报到页手机号自动填充且不可编辑
6. 能正常选择身份证照片并显示预览
7. 图片格式或大小不合要求时，系统给出明确提示
8. 提交成功后，后台学生记录中包含身份证照片的有效访问链接
9. 图片文件保存在服务器指定目录下，不会因重启丢失
10. 未登录状态下直接访问提交接口被拒绝
11. 网页版和小程序版均通过上述测试

# 二、系统架构设计
## 2.1 总体架构
### 2.1.1 架构模式
采用经典三层架构，在 2.0 基础上扩展身份认证和文件上传能力：

**表现层**：微信小程序 + 网页端，提供手机号验证页面和表单输入界面。

**业务逻辑层**：Django 框架，编写验证码发送、校验、登录接口，以及信息提交与文件上传接口。

**数据持久层**：SQLite 数据库，建立验证码记录表和学生信息表，身份证照片存储在服务器本地媒体目录。

# 三、技术实现方案
## 3.1 数据库设计
### 3.1.1 数据模型关系
```
验证码记录表 ──(查询)──→ 手机号
                           ↓
学生信息表 ←──(关联)── 手机号（自动填充）
    ↓
报到记录表（一对一关联）
```

### 3.1.2 核心表结构说明

#### 验证码记录表（VerificationCode）
| 字段名称 | 类型 | 说明 |
|----------|------|------|
| phone | CharField | 手机号，11 位 |
| code | CharField | 4 位随机数字验证码 |
| created_at | DateTimeField | 验证码发送时间 |
| is_used | BooleanField | 是否已使用，默认 False |
| expires_at | DateTimeField | 过期时间（发送时间 + 5 分钟） |

#### 学生信息表（StudentInfo）
| 字段名称 | 类型 | 说明 |
|----------|------|------|
| name | CharField | 学生姓名 |
| student_id | CharField | 学号，唯一标识 |
| phone | CharField | 手机号，从 Session 自动获取 |
| id_card_photo | ImageField | 身份证照片路径 |
| created_at | DateTimeField | 记录创建时间 |

#### 报到记录表（RegistrationRecord）
| 字段名称 | 类型 | 说明 |
|----------|------|------|
| student | OneToOneField | 关联学生信息 |
| registered_at | DateTimeField | 报到时间 |

## 3.2 接口设计
### 3.2.1 发送验证码接口
| 项目 | 说明 |
|------|------|
| 接口地址 | `/api/send_code/` |
| 请求方式 | POST |
| 请求参数 | `{ "phone": "13800138000" }` |
| 响应数据 | `{ "success": true, "message": "验证码已发送" }` |

**处理逻辑**：
1. 校验手机号是否为 11 位中国大陆手机号格式
2. 检查该手机号在 60 秒内是否已经请求过验证码
3. 生成 4 位随机数字验证码
4. 将手机号、验证码、发送时间存入验证码记录表
5. 在控制台中打印验证码内容
6. 返回发送成功提示

### 3.2.2 验证码校验与登录接口
| 项目 | 说明 |
|------|------|
| 接口地址 | `/api/verify_code/` |
| 请求方式 | POST |
| 请求参数 | `{ "phone": "13800138000", "code": "1234" }` |
| 响应数据 | `{ "success": true, "message": "登录成功", "data": { "phone": "13800138000" } }` |

**处理逻辑**：
1. 根据手机号查询验证码记录表中最新一条未使用的记录
2. 校验验证码是否匹配
3. 校验验证码是否在有效期（5 分钟）内
4. 校验通过后将该记录标记为已使用
5. 建立登录 Session，标记用户已验证状态
6. 返回登录成功信息

### 3.2.3 信息提交与身份证上传接口
| 项目 | 说明 |
|------|------|
| 接口地址 | `/api/register_with_info/` |
| 请求方式 | POST |
| 请求参数 | FormData：name、student_id、id_card_photo（文件） |
| 响应数据 | `{ "success": true, "message": "报到成功", "data": { ... } }` |

**处理逻辑**：
1. 校验登录状态，未登录则拒绝请求
2. 校验文本字段是否完整
3. 校验身份证照片是否为允许的图片格式（.jpg、.png）
4. 限制图片文件大小（5MB 以内）
5. 将图片保存到媒体目录，存储路径
6. 将手机号、姓名、学号、身份证照片路径一同写入学生信息表
7. 创建报到记录
8. 返回成功响应

## 3.4 安全设计
### 3.4.1 验证码安全
- 60 秒内不允许重复发送验证码
- 验证码有效期 5 分钟，过期自动失效
- 验证码使用后立即标记为已使用

### 3.4.2 文件上传安全
- 限制允许的图片格式（.jpg、.png）
- 限制文件大小（5MB 以内）
- 文件存储在服务器本地，避免路径遍历攻击

### 3.4.3 接口安全
- 信息提交接口需要登录状态
- 未登录访问返回 401 未授权响应
- 开发阶段关闭 CSRF 保护（后续生产环境需开启）

## 3.5 性能优化
- 验证码查询使用索引（手机号字段）
- 图片上传使用异步处理（可选）
- 媒体文件使用 Nginx 静态服务（生产环境）

# 四、运维部署方案
## 4.1 开发环境
| 配置项 | 要求 |
|--------|------|
| 操作系统 | Windows 10 或 macOS |
| Python 版本 | 3.10 及以上 |
| 微信开发者工具 | 最新稳定版 |
| 代码编辑器 | VS Code（推荐） |

## 4.2 本地部署流程说明
1. 安装 Python 环境并确认版本符合要求
2. 使用包管理工具安装 Django 框架
3. 通过 Django 命令行工具创建项目目录结构
4. 在项目内创建新的应用模块用于处理接口逻辑
5. 配置媒体文件存储路径和访问 URL
6. 在模型文件中定义验证码记录表和学生信息表结构
7. 执行数据库迁移命令，使模型定义在 SQLite 数据库中生效
8. 编写验证码发送、校验登录、信息提交上传接口
9. 在项目路由配置中将接口路径与视图函数绑定
10. 通过命令行创建管理员账号，并将模型注册到管理后台
11. 启动 Django 内置的开发服务器
12. 在微信开发者工具中创建小程序项目，实现验证页面和表单页面
13. 在开发者工具中勾选"不校验合法域名"选项以允许访问本地服务
14. 点击小程序验证登录，填写表单并提交，确认数据链路通畅

## 4.3 后续生产环境建议
- 将数据库切换为 PostgreSQL
- 采用 Nginx + Gunicorn 部署 Django 应用
- 为域名申请 SSL 证书并启用 HTTPS
- 将媒体文件迁移至云存储服务（如阿里云 OSS、腾讯云 COS）
- 接入真实短信服务（如阿里云短信、腾讯云短信）

# 十、项目心得体会
## 核心收获
通过 3.0 版本的升级，我们同时引入了手机号验证码登录和身份证照片上传两个重要功能，使系统具备了完整的身份认证和材料采集能力。在技术选型上，Django 的 Session 机制和媒体文件处理能力为我们提供了坚实的基础，让我们能够以最小的代码量实现复杂的功能。

## 设计反思
本版本在功能设计上更加注重用户体验和系统安全性。通过手机号验证码登录确保了身份的真实性，通过身份证照片上传提供了身份证明材料。在技术实现上，我们合理利用了 Django 的内置功能，避免了重复造轮子。同时，通过合并接口的方式，将信息提交和图片上传整合为一个接口，简化了前端调用逻辑。

## 未来展望
在具备完整身份认证和材料采集能力的基础上，我们将逐步引入管理员审核、宿舍分配等功能。随着功能的不断完善，系统将逐渐演进为一套能够真正服务新生报到场景的智慧迎新平台。
