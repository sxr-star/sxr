# 新生报到项目（只跑通版）   苏鑫茹

# 一、执行摘要
## 1. 核心目标
一句话定义：建立一条从微信小程序到后端数据库的完整数据通道。新生在小程序端触发一个简单操作，数据能够成功传输到 Django 后端并存入数据库，管理员在后台能够直观地看到这条记录。

## 2. 功能范围界定
### 必做功能
小程序端提供一个触发按钮，后端提供一个数据接收接口，数据库建立一张记录表，管理员后台可查看记录。

### 不实现功能
❌ 不做微信登录（采用固定标识代替用户身份）
❌ 不做信息填写表单
❌ 不做图片上传
❌ 不做审核与宿舍分配
❌ 不做任何复杂业务逻辑

# 二、系统架构设计
## 2.1 总体架构
### 2.1.1 架构模式
采用经典三层架构，仅保留核心数据通路：
表现层：微信小程序，提供一个操作按钮与简单提示。
业务逻辑层：Django 框架，仅编写一个用于接收请求并写入数据的接口。
数据持久层：SQLite 数据库，仅建立一张用于存储操作记录的简单数据表。

## 2.2 技术架构
### 2.2.1 技术栈选型
| 层次 | 技术选型 | 选型理由 |

# 三、技术实现方案
## 3.1 数据库设计
### 3.1.1 数据模型关系
本阶段仅存在一张独立的数据表，不涉及表间关联。

### 3.1.2 核心表结构说明
表名称为 "操作记录表"，包含三个字段：
- 自增主键：唯一标识每条记录，整数类型。
- 备注信息：字符串类型，用于简单说明数据来源，可设置默认值。
- 创建时间：日期时间类型，在记录生成时自动填入当前时间。

该表的作用是承载从小程序端传输过来的每一次操作痕迹，作为验证数据链路通畅的直接证据。

## 3.2 接口设计
系统仅需暴露一个接口，说明如下：
- 接口地址：`/api/register/`
- 请求方法：POST
- 功能描述：接收来自小程序的请求，在数据库操作记录表中新增一行数据。
- 请求参数：本阶段无需传递任何参数，接口接收到请求即执行写入。
- 返回数据：返回一个固定格式的 JSON，包含表示操作成功的布尔值字段。

## 3.3 安全设计
开发调试阶段暂时关闭框架的跨站请求伪造保护，以便小程序能够顺利发送请求。由于不涉及真实用户信息，暂无数据加密与隐私保护需求。后续迭代将逐步加入身份认证与输入校验机制。

## 3.4 性能优化
单次请求仅执行一条插入语句，响应时间极短。数据库字段设计精简，无需额外索引优化。利用 Django 开发服务器即可满足调试阶段的并发需求。

# 四、运维部署方案
## 4.1 开发环境
| 配置项 | 要求 |
|--------|------|
| 操作系统 | Windows 10 或 macOS |
| Python 版本 | 3.10 及以上 |
| 微信开发者工具 | 最新稳定版 |
| 代码编辑器 | VS Code（推荐） |

## 4.2 本地部署流程说明
1. 在本地计算机安装 Python 环境并确认版本符合要求。
2. 使用包管理工具安装 Django 框架。
3. 通过 Django 命令行工具创建项目目录结构。
4. 在项目内创建一个新的应用模块用于处理接口逻辑。
5. 在应用目录下的模型文件中定义操作记录表的结构。
6. 执行数据库迁移命令，使模型定义在 SQLite 数据库中生效。
7. 在视图文件中编写一个函数，该函数调用模型的方法向表中插入数据，并返回成功响应。
8. 在项目路由配置中将接口路径与上述视图函数绑定。
9. 通过命令行创建管理员账号，并将操作记录表注册到管理后台。
10. 启动 Django 内置的开发服务器。
11. 在微信开发者工具中创建小程序项目，在页面中放置按钮并绑定事件，事件内调用网络请求 API 指向本地服务器的接口地址。
12. 在开发者工具中勾选 "不校验合法域名" 选项以允许访问本地服务。
13. 点击小程序按钮，随后登录 Django 管理后台查看操作记录表，确认新记录出现即表示链路通畅。

## 4.3 后续生产环境建议
待流程验证完毕后，可将数据库切换为更适合生产环境的 PostgreSQL。采用 Nginx 配合 Gunicorn 的方式部署 Django 应用。为域名申请 SSL 证书并启用 HTTPS 加密通信。将静态文件与媒体文件托管至云存储服务。

# 九、后续迭代方向
在确保基础数据通道完全通畅之后，按优先级逐步增加以下功能：
1. **用户身份标识**：接入微信登录，获取真实的用户唯一标识。
2. **信息采集表单**：在小程序端增加输入框，采集姓名、学号等字段并随请求发送。
3. **图片上传能力**：支持身份证和录取通知书照片的选取与上传。
4. **审核状态管理**：在数据库表中增加审核状态字段，管理员可通过后台进行审批。
5. **宿舍信息展示**：管理员填写宿舍号后，新生可在小程序端查询到分配结果。

# 十、项目心得体会
## 核心收获
通过本次极简流程框架的搭建，我深刻认识到一个完整系统的最小单元是 "前端触发、后端处理、数据库存储"。这个看似简单的链路是后续所有复杂功能的基石。技术选型上，Django 加 SQLite 加微信小程序原生开发的组合，将环境配置和代码编写的工作量降到了最低，让我能够完全专注于理解数据流动的方向和各个环节的职责。

## 设计反思
本方案将功能压缩到只有一个按钮和一个接口的程度，贯彻了 "先跑通、再优化" 的工程实践原则。这种极简设计让我在最短时间内获得了正向反馈，建立了继续开发的信心。同时我也意识到，这样的方案虽然验证了核心链路，但缺少必要的异常处理和输入校验，在真实使用场景下还需进一步完善。

## 未来展望
在这条通畅的 "数据管道" 基础上，我将遵循由简入繁的迭代思路，逐步添加用户身份、表单采集、文件上传和业务流程控制。随着功能的累加，系统的实用价值将逐渐显现，最终演进为一套能够真正服务新生报到场景的智慧迎新平台。
