# 新生报到系统 3.0 - 需求报告

## 一、升级目标

在 2.0 版本"个人信息采集"的基础上，同时增加手机号验证码登录和身份证照片上传两个功能。新生在报到前需先通过手机号验证身份，填表时需上传身份证照片作为身份证明材料。系统将手机号与学生身份绑定，身份证照片保存在服务器本地，管理员可在后台查看。

**一句话目标**：学生先用手机号验证登录，然后填表、上传身份证照片、提交报到。

---

## 二、新增功能范围

### ✅ 新增功能

| 功能 | 描述 |
|------|------|
| 手机号验证码发送 | 开发阶段用控制台打印模拟，生产环境替换为真实短信服务 |
| 验证码校验与登录 | 验证通过后建立登录 Session，标记用户已验证状态 |
| 身份证照片上传 | 支持常见图片格式，限制文件大小 |
| 手机号与学生信息自动关联 | 提交报到时自动将当前登录用户的手机号填入学生信息 |

### ✅ 保留功能

| 功能 | 描述 |
|------|------|
| 个人信息填写表单 | 姓名、学号、手机号（2.0 已有） |
| 表单数据持久化 | 数据存入数据库，与报到记录关联 |
| 多端同步更新 | 网页版和小程序端同步支持 |

### ❌ 暂不加入功能

- 管理员审核
- 宿舍分配

## 三、功能需求详细说明

### 3.1 手机号验证码登录

#### 3.1.1 验证码机制

| 项目 | 说明 |
|------|------|
| 验证码格式 | 4 位随机数字 |
| 有效期 | 5 分钟 |
| 发送频率限制 | 60 秒内不允许重复发送 |
| 开发阶段 | 后端控制台打印验证码 |
| 生产环境 | 替换为真实短信接口 |

#### 3.1.2 登录流程

1. 用户输入手机号，点击"发送验证码"
2. 后端校验手机号格式，生成验证码并存储，打印到控制台
3. 用户输入收到的验证码，点击"验证并登录"
4. 后端校验验证码是否正确、是否过期、是否已使用
5. 验证通过后建立 Session，标记用户已验证状态
6. 前端保存登录状态，跳转到报到页面

### 3.2 身份证照片上传

#### 3.2.1 图片要求

| 项目 | 说明 |
|------|------|
| 支持格式 | .jpg、.png 等常见图片格式 |
| 文件大小 | 限制在 5MB 以内 |
| 存储方式 | 保存在服务器本地媒体目录 |
| 访问方式 | 通过媒体文件服务访问 |

#### 3.2.2 上传流程

1. 用户在报到页面点击"选择身份证照片"
2. 选择图片文件后显示预览缩略图
3. 提交时前端校验图片格式和大小
4. 后端接收图片文件，保存到媒体目录
5. 将图片路径与其他信息一同存入数据库

### 3.3 个人信息填写表单

#### 3.3.1 表单字段

| 字段名称 | 类型 | 必填 | 说明 |
|----------|------|------|------|
| 姓名 | 字符串 | 是 | 学生真实姓名 |
| 学号 | 字符串 | 是 | 学生学号，唯一标识 |
| 手机号 | 字符串 | 是 | 自动填充当前登录用户的手机号，不可编辑 |
| 身份证照片 | 文件 | 是 | 学生身份证照片 |

#### 3.3.2 表单交互要求

- 需要登录状态才能访问，未登录自动跳转到验证页面
- 手机号字段自动填充当前登录用户的手机号，且设为不可编辑
- 增加身份证照片选择区域，支持图片预览
- 提交按钮将文本数据和图片文件一并发送
- 前端需做非空校验和图片格式校验

### 3.4 项目配置调整

#### 3.4.1 媒体文件配置

| 配置项 | 说明 |
|--------|------|
| 媒体根目录 | 定义身份证照片的存储路径 |
| 媒体访问 URL | 配置媒体文件的访问地址 |
| 开发服务器 | 确保开发服务器能够正确提供媒体文件服务 |

### 3.5 数据库模型升级

#### 3.5.1 验证码记录表

| 字段名称 | 类型 | 说明 |
|----------|------|------|
| 手机号 | 字符串 | 用户输入的手机号 |
| 验证码 | 字符串 | 4 位随机数字 |
| 发送时间 | 日期时间 | 验证码发送时间 |
| 是否已使用 | 布尔值 | 标记验证码是否已被使用 |
| 过期时间 | 日期时间 | 验证码失效时间 |

#### 3.5.2 学生信息表（扩展）

| 字段名称 | 类型 | 说明 |
|----------|------|------|
| 姓名 | 字符串 | 学生真实姓名 |
| 学号 | 字符串 | 学生学号，唯一标识 |
| 手机号 | 字符串 | 学生联系电话 |
| 身份证照片路径 | 字符串 | 身份证照片的存储路径 |
| 创建时间 | 日期时间 | 记录创建时间 |

### 3.6 后端接口设计

#### 3.6.1 发送验证码接口

| 项目 | 说明 |
|------|------|
| 接口地址 | `/api/send_code/` |
| 请求方式 | POST |
| 请求参数 | 手机号 |
| 处理逻辑 | 校验手机号格式 → 检查 60 秒内是否重复发送 → 生成验证码 → 存储 → 打印控制台 → 返回成功 |

#### 3.6.2 验证码校验与登录接口

| 项目 | 说明 |
|------|------|
| 接口地址 | `/api/verify_code/` |
| 请求方式 | POST |
| 请求参数 | 手机号、验证码 |
| 处理逻辑 | 查询最新未使用记录 → 校验验证码 → 校验有效期 → 标记已使用 → 建立 Session → 返回成功 |

#### 3.6.3 信息提交与身份证上传接口（合并接口）

| 项目 | 说明 |
|------|------|
| 接口地址 | `/api/register_with_info_v3/` |
| 请求方式 | POST |
| 请求参数 | 姓名、学号、身份证照片文件 |
| 特殊说明 | 手机号从当前登录 Session 中自动获取，无需前端传递 |
| 处理逻辑 | 检查登录状态 → 校验文本字段 → 校验图片格式和大小 → 保存图片 → 存储路径 → 写入数据库 → 返回成功 |

### 3.7 前端改动

#### 3.7.1 网页版

- 新增手机号验证页面，包含手机号输入框、发送验证码按钮、验证码输入框、登录按钮
- 实现发送验证码按钮 60 秒倒计时效果
- 验证成功后跳转到报到页，手机号自动填充且锁定
- 报到页增加身份证照片选择控件和预览区域
- 使用 FormData 封装文本和图片数据一并提交

#### 3.7.2 小程序版

- 新增登录页面，包含手机号输入框、发送验证码按钮、验证码输入框、登录按钮
- 实现发送验证码按钮 60 秒倒计时效果
- 验证成功后保存登录状态到本地存储，跳转报到页
- 报到页增加身份证照片选择按钮，调用微信选择图片接口
- 实现图片预览，使用微信上传文件接口发送数据

### 3.8 管理后台调整

- 在学生信息列表中可以直观看到姓名、学号、手机号、报到时间
- 身份证照片字段在后台中显示为可点击的链接，点击可查看原图
- 验证码记录表注册到后台，方便调试时排查问题

## 七、注意事项

| 序号 | 注意事项 | 说明 |
|------|----------|------|
| 1 | 频率限制 | 验证码接口需做好频率限制，防止被恶意调用 |
| 2 | 短信服务 | 开发阶段使用控制台打印验证码，生产环境务必替换为真实短信服务 |
| 3 | Session 有效期 | Session 有效期可适当设置长一些，避免报到中途登录态失效 |
| 4 | 文件管理 | 身份证图片上传后注意定期清理测试数据，避免占用过多磁盘空间 |
| 5 | 小程序配置 | 小程序上传图片时需在项目配置中声明上传文件合法域名 |
| 6 | 生产环境 | 部署到生产环境时建议将媒体文件迁移到云存储服务 |

## 八、项目心得体会

### 核心收获

通过 3.0 版本的升级，我们同时引入了手机号验证码登录和身份证照片上传两个重要功能，使系统具备了完整的身份认证和材料采集能力。这一改进不仅提升了系统的安全性，也为后续的审核流程打下了坚实的基础。在技术实现上，我们采用了合并接口的方式，将信息提交和图片上传整合为一个接口，简化了前端调用逻辑。

### 设计反思

本版本在功能设计上更加注重用户体验，通过手机号验证码登录确保了身份的真实性，通过身份证照片上传提供了身份证明材料。在技术实现上，我们合理利用了 Django 的媒体文件处理机制，确保了图片的安全存储和访问。同时，通过 Session 机制实现了登录状态的保持，为用户提供了流畅的报到体验。

### 未来展望

在具备完整身份认证和材料采集能力的基础上，我们将逐步引入管理员审核、宿舍分配等功能。随着功能的不断完善，系统将逐渐演进为一套能够真正服务新生报到场景的智慧迎新平台。
