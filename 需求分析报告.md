# 新生报到系统 4.0 - 需求分析报告

## 一、引言

### 1.1 项目背景

随着高校信息化建设的推进，传统的新生报到流程已难以满足高效、便捷的服务需求。本系统旨在通过数字化手段，实现新生报到流程的线上化、智能化，提升报到效率和服务质量。

### 1.2 文档目的

本需求分析报告详细描述新生报到系统 4.0 版本的功能需求、非功能需求、用例模型和用户故事，为系统设计、开发和测试提供明确的需求依据。4.0 版本在 3.0 版本"手机号验证登录 + 信息采集 + 身份证上传"的基础上，新增管理员审核和报到数据统计看板两个功能模块，形成完整的报到管理闭环。

### 1.3 版本历史

| 版本 | 日期 | 作者 | 变更说明 |
|------|------|------|----------|
| 1.0 | 2026-04-24 | 苏鑫茹 | 初始版本 |
| 2.0 | 2026-04-27 | 苏鑫茹 | 新增信息采集功能需求 |
| 3.0 | 2026-05-19 | 苏鑫茹 | 新增身份认证与身份证上传功能 |
| 4.0 | 2026-05-19 | 苏鑫茹 | 新增管理员审核、宿舍分配与数据统计看板功能 |

### 1.4 一句话目标

学生提交报到 → 管理员审核分配宿舍 → 后台统计看板实时展示报到数据，形成完整的报到管理闭环。

## 二、功能需求

### 2.1 功能需求总览

| 功能模块 | 功能点 | 需求来源 | 优先级 | 版本 |
|----------|--------|----------|--------|------|
| 身份认证 | 手机号验证码登录 | 业务安全要求 | ⭐⭐⭐ | 3.0 |
| 信息采集 | 个人信息填写 | 报到业务需求 | ⭐⭐⭐ | 3.0 |
| 材料上传 | 身份证正反面照片上传 | 身份核验要求 | ⭐⭐⭐ | 3.0 |
| 报到状态管理 | 待审核/审核通过/审核驳回 | 审核流程需求 | ⭐⭐⭐ | 4.0 |
| 管理员审核 | 通过/驳回操作 | 后台管理需求 | ⭐⭐⭐ | 4.0 |
| 宿舍分配 | 管理员分配宿舍号 | 报到业务需求 | ⭐⭐⭐ | 4.0 |
| 状态查询 | 新生查看审核结果与宿舍号 | 新生端需求 | ⭐⭐⭐ | 4.0 |
| 数据统计看板 | 报到进度实时统计 | 后台管理需求 | ⭐⭐⭐ | 4.0 |
| 数据管理 | 学生信息管理 | 后台管理需求 | ⭐⭐⭐ | 3.0 |

### 2.2 身份认证模块（保留 3.0）

#### 2.2.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-AUTH-001 | 支持手机号输入 | 用户输入11位中国大陆手机号 |
| FR-AUTH-002 | 验证码发送 | 点击按钮发送4位数字验证码 |
| FR-AUTH-003 | 发送频率限制 | 60秒内不可重复发送 |
| FR-AUTH-004 | 验证码校验 | 验证用户输入的验证码正确性 |
| FR-AUTH-005 | 登录状态保持 | 验证通过后建立Session |
| FR-AUTH-006 | 过期处理 | 验证码5分钟内有效 |

#### 2.2.2 业务规则

- 验证码格式：4位随机数字
- 有效期：5分钟
- 发送间隔：60秒
- 验证码使用后立即失效

### 2.3 信息采集模块（保留 3.0）

#### 2.3.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-INFO-001 | 姓名输入 | 学生真实姓名 |
| FR-INFO-002 | 学号输入 | 学生学号，唯一标识 |
| FR-INFO-003 | 手机号自动填充 | 从Session获取，不可编辑 |
| FR-INFO-004 | 表单验证 | 必填字段校验 |

#### 2.3.2 业务规则

- 学号必须唯一
- 手机号从登录Session自动获取
- 表单提交前必须完成验证

### 2.4 材料上传模块（保留 3.0）

#### 2.4.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-UPLOAD-001 | 身份证正面上传 | 人像面照片 |
| FR-UPLOAD-002 | 身份证反面上传 | 国徽面照片 |
| FR-UPLOAD-003 | 图片预览 | 选择后显示缩略图 |
| FR-UPLOAD-004 | 格式校验 | 仅支持.jpg/.png |
| FR-UPLOAD-005 | 大小限制 | 单张不超过5MB |
| FR-UPLOAD-006 | 重复检测 | 防止上传同一张照片 |
| FR-UPLOAD-007 | 正反面识别 | 自动识别并纠正顺序 |

#### 2.4.2 业务规则

- 必须同时上传正反两面
- 两张照片不能相同
- 系统自动识别正反面，顺序错误时自动纠正

### 2.5 报到状态管理模块（新增 4.0）

#### 2.5.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-STATUS-001 | 默认状态设置 | 新生提交报到后，状态自动设为"待审核" |
| FR-STATUS-002 | 状态枚举值 | 可选值：待审核、审核通过、审核驳回 |
| FR-STATUS-003 | 状态变更通知 | 状态变更后，前端可实时查询最新状态 |

#### 2.5.2 业务规则

- 新生首次提交报到信息后，状态默认为"待审核"
- 状态只能由管理员通过审核操作变更
- 审核驳回后，学生可重新提交信息，状态重置为"待审核"

### 2.6 管理员审核模块（新增 4.0）

#### 2.6.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-REVIEW-001 | 审核通过操作 | 管理员可将学生状态设为"审核通过" |
| FR-REVIEW-002 | 审核驳回操作 | 管理员可将学生状态设为"审核驳回" |
| FR-REVIEW-003 | 驳回原因填写 | 审核驳回时，管理员必须填写驳回原因 |
| FR-REVIEW-004 | 宿舍号分配 | 审核通过时，管理员可填写宿舍号（选填） |
| FR-REVIEW-005 | 权限校验 | 仅管理员可执行审核操作 |

#### 2.6.2 业务规则

- 审核通过时，宿舍号字段可填写，非必填
- 审核驳回时，驳回原因字段必填，长度不超过500字
- 审核操作需记录操作人和操作时间
- 非管理员账号调用审核接口应返回403错误

### 2.7 状态查询模块（新增 4.0）

#### 2.7.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-QUERY-001 | 审核状态显示 | 新生可查看当前审核状态 |
| FR-QUERY-002 | 驳回原因显示 | 审核驳回时，显示管理员填写的驳回原因 |
| FR-QUERY-003 | 宿舍号显示 | 审核通过时，显示分配的宿舍号 |
| FR-QUERY-004 | 重新提交入口 | 审核驳回时，提供重新提交按钮 |

#### 2.7.2 业务规则

- 状态查询接口需校验登录状态，未登录拒绝访问
- 仅可查询当前登录用户自己的审核状态
- 审核驳回时，点击重新提交跳转回报到页面，保留原有填写信息

### 2.8 数据统计看板模块（新增 4.0）

#### 2.8.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-DASH-001 | 总提交人数统计 | 统计学生信息表总记录数 |
| FR-DASH-002 | 待审核人数统计 | 统计状态为"待审核"的记录数 |
| FR-DASH-003 | 审核通过人数统计 | 统计状态为"审核通过"的记录数 |
| FR-DASH-004 | 审核驳回人数统计 | 统计状态为"审核驳回"的记录数 |
| FR-DASH-005 | 通过率计算 | 通过率 = 审核通过人数 / 总提交人数 × 100% |
| FR-DASH-006 | 权限校验 | 仅管理员可访问统计数据接口 |

#### 2.8.2 业务规则

- 统计数据为实时查询，每次刷新页面重新统计
- 通过率为百分比，保留两位小数
- 非管理员账号调用统计接口应返回403错误
- 点击看板卡片可跳转到对应状态的学生信息筛选列表

### 2.9 数据管理模块（优化 4.0）

#### 2.9.1 功能需求

| 需求编号 | 需求描述 | 说明 |
|----------|----------|------|
| FR-MGMT-001 | 学生信息列表 | 查看所有报到学生 |
| FR-MGMT-002 | 身份证查看 | 点击链接查看原图 |
| FR-MGMT-003 | 验证码记录 | 查看验证码发送记录 |
| FR-MGMT-004 | 审核状态筛选 | 按审核状态筛选学生列表 |
| FR-MGMT-005 | 审核状态列显示 | 列表页显示每个学生当前审核状态 |
| FR-MGMT-006 | 自定义审核操作 | 列表页提供审核通过/驳回按钮 |

### 8.2 新增接口详细设计

#### 8.2.1 状态查询接口

| 项目 | 内容 |
|------|------|
| 请求方式 | GET |
| 请求路径 | /api/review_status/ |
| 认证要求 | 需要登录（Session） |
| 请求参数 | 无（通过Session识别当前用户） |
| 成功响应 | `{"success": true, "data": {"review_status": "待审核", "reject_reason": null, "dormitory_number": null}}` |
| 失败响应 | `{"success": false, "message": "请先登录"}` |
| 处理逻辑 | 1.校验登录状态<br>2.根据Session中的手机号查询学生信息<br>3.返回审核状态、驳回原因、宿舍号 |

#### 8.2.2 管理员审核接口

| 项目 | 内容 |
|------|------|
| 请求方式 | POST |
| 请求路径 | /api/admin/review/ |
| 认证要求 | 需要管理员权限 |
| 请求体 | `{"student_id": "学号", "action": "approve/reject", "reject_reason": "驳回原因（驳回时必填）", "dormitory_number": "宿舍号（通过时选填）"}` |
| 成功响应 | `{"success": true, "message": "审核成功"}` |
| 失败响应 | `{"success": false, "message": "权限不足"}` 或 `{"success": false, "message": "驳回原因不能为空"}` |
| 处理逻辑 | 1.校验管理员权限<br>2.根据学号找到学生记录<br>3.更新审核状态<br>4.若通过，更新宿舍号<br>5.若驳回，更新驳回原因 |

#### 8.2.3 统计数据接口

| 项目 | 内容 |
|------|------|
| 请求方式 | GET |
| 请求路径 | /api/admin/statistics/ |
| 认证要求 | 需要管理员权限 |
| 请求参数 | 无 |
| 成功响应 | `{"success": true, "data": {"total": 100, "pending": 30, "approved": 60, "rejected": 10, "pass_rate": "60.00%"}}` |
| 失败响应 | `{"success": false, "message": "权限不足"}` |
| 处理逻辑 | 1.校验管理员权限<br>2.查询学生信息表统计各状态人数<br>3.计算通过率<br>4.返回统计数据 |

## 十、约束与限制

### 10.1 技术约束

- 开发阶段使用控制台打印验证码，生产环境需接入短信服务
- 图片存储在本地媒体目录，生产环境建议迁移至云存储
- 数据库使用SQLite，生产环境建议切换至PostgreSQL
- 管理员权限利用Django自带的用户权限系统实现

### 10.2 业务约束

- 学号必须唯一，不允许重复报到
- 身份证照片必须同时上传正反两面
- 验证码有效期为5分钟，使用后立即失效
- 审核驳回后学生重新提交时，建议保留原有填写信息
- 宿舍号分配目前为手动填写，后续版本可扩展为自动分配算法
- 统计看板数据实时查询，后期数据量增大可加入缓存机制

### 10.3 暂不实现功能

| 功能 | 说明 | 计划版本 |
|------|------|----------|
| 消息推送通知 | 审核结果推送给新生 | 后续版本 |
| 批量导入导出 | 学生数据批量操作 | 后续版本 |
| 数据可视化图表 | 报到数据图表展示 | 后续版本 |
| 校园导览功能 | 地图集成、地点标注、导航 | 5.0 |
